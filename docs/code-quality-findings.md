## Context

from the source repository [Shuozeli/fbsviewer-lib](https://github.com/Shuozeli/fbsviewer-lib).

Files present:

The only auditable code is the inline `<script>` blocks in `index.html` (lines 50-76).

## 1. Silent Failures / Unsafe Patterns

### 1.1 MutationObserver never disconnects -- DONE
- **Location:** `index.html:54-57` (MutationObserver callback)
- **Problem:** The `MutationObserver` that hides the loading spinner never calls
  `observer.disconnect()` after hiding the element. Once the WASM canvas initializes
  and the spinner is hidden, the observer continues to fire on every subsequent canvas
  attribute mutation (e.g., resize events, WebGL state changes). This is a minor
  resource leak that causes unnecessary callback invocations for the lifetime of the page.
- **Fix:** Call `observer.disconnect()` inside the callback after hiding the loading element:
  ```js
  var observer = new MutationObserver(function() {
      document.getElementById("loading").style.display = "none";
      observer.disconnect();
  });
  ```
- **Severity:** Low. The callback is trivial and the leak is small, but this is a
  correctness issue -- the observer serves no purpose after the first trigger.

### 1.2 No null-check on getElementById inside observer callback -- DONE
- **Location:** `index.html:55`
- **Problem:** `document.getElementById("loading").style.display = "none"` will throw a
  `TypeError` if the element is absent. While the element is defined in the static HTML
  and unlikely to be missing, the callback fires on *every* attribute mutation (due to
  issue 1.1), so if the element were ever removed from the DOM, subsequent triggers would
  throw.
- **Fix:** Guard with a null check, or (better) fix issue 1.1 so the callback only fires once:
  ```js
  var loading = document.getElementById("loading");
  if (loading) loading.style.display = "none";
  observer.disconnect();
  ```
- **Severity:** Low (cosmetic). The static HTML guarantees the element exists, but
  defensive coding is preferred.

## 2. No-op / Unnecessary Code

### 2.1 Global binding assignment may be unnecessary -- SKIPPED
- **Reason:** This is Trunk-generated boilerplate. The `TrunkApplicationStarted` event and `window.wasmBindings` assignment are part of the Trunk build output pattern. Removing it could break the Trunk lifecycle contract. Since this is a deployment artifact repo (not the source), changes here would be overwritten on the next build. The fix belongs in Trunk configuration in the source repo.
- **Location:** `index.html:72` (`window.wasmBindings = bindings`)
- **Problem:** The WASM bindings are assigned to `window.wasmBindings`, polluting the
  global namespace. This appears to be generated by Trunk (the Rust WASM bundler) and
  may have been useful during development for debugging, but in a production deployment
  it exposes internal WASM bindings to any script on the page.
- **Fix:** If this is Trunk-generated boilerplate, it may not be easily removable without
  changing the Trunk build configuration. If manual, remove the line:
  ```js
  // Remove this line unless needed for external integrations:
  window.wasmBindings = bindings;
  ```
- **Severity:** Low (cosmetic). No security risk since the page has no third-party scripts,
  but it is unnecessary namespace pollution.

## 3. Missing Abstractions / Infrastructure

### 3.1 No error handling for WASM initialization failure -- DONE
- **Location:** `index.html:68-75` (module script block)
- **Problem:** The WASM module is loaded with a bare `await init(...)` on line 69. If
  the `.wasm` file fails to load (network error, corrupt file, unsupported browser), the
  promise rejection is unhandled and the user sees the loading spinner indefinitely with
  no error message. There is no `try/catch` and no `.catch()` handler.
- **Fix:** Wrap the initialization in a try/catch that shows a user-facing error:
  ```js
  try {
      const wasm = await init({
          module_or_path: './flatbuf-visualizer-52e17ab52898f434_bg.wasm'
      });
      window.wasmBindings = bindings;
      dispatchEvent(new CustomEvent("TrunkApplicationStarted", {detail: {wasm}}));
  } catch (e) {
      var loading = document.getElementById("loading");
      if (loading) {
          loading.innerHTML = '<p>Failed to load WebAssembly module.</p>'
              + '<p style="font-size:14px;color:#999">' + e.message + '</p>';
      }
      console.error("WASM init failed:", e);
  }
  ```
- **Severity:** Medium. Users on unsupported browsers or with network issues get a
  perpetual spinner with no feedback. This is the most impactful finding in this audit.

## 4. Inconsistent Patterns

### 4.1 Mixed var and const in inline scripts -- DONE
- **Location:** `index.html:53` (`var canvas`), `index.html:54` (`var observer`)
- **Problem:** The first script block uses `var` (lines 53-54) while the second script
  block uses `const` and `await` (ES module syntax, lines 68-75). This is a minor
  inconsistency. The first block uses `var` likely because it is a classic script (not
  `type="module"`), but `let`/`const` are still valid in classic scripts and would be
  more consistent.
- **Fix:** Change `var` to `const` in the first script block:
  ```js
  const canvas = document.getElementById("the_canvas_id");
  const observer = new MutationObserver(function() { ... });
  ```
- **Severity:** Low (cosmetic).

## 5. Recommendation

The meaningful code quality work should target the **source repository**
([Shuozeli/fbsviewer-lib](https://github.com/Shuozeli/fbsviewer-lib)), which contains
the actual Rust application code, build configuration, and tests. This deployment
repository holds only build artifacts and a thin HTML wrapper.

Of the findings above, **issue 3.1** (no WASM init error handling) is the only one with
user-facing impact. Issues 1.1, 1.2, 2.1, and 4.1 are low-severity polish items.
